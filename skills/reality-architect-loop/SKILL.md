# Reality Architect Loop

Take a vision and run it through one complete loop that ends in something real. This is the half that manifestation usually skips: the build. Manifestation Master is the inner game (set a vision, feel it, hold attention); Reality Architect adds the system that ships it. Based on the open [Reality Architect](https://github.com/frankxai/realityarchitect) method.

## When to use

The user wants to turn a vision into reality, build a system around a goal, or move from "I keep visualizing this" to "this exists now."

## The five moves

Run them in order. Each depends on the one before.

1. **See — capture the context.** What does the agent (and the user) need to know to pursue this? Pull the vision, the intention spec (use the `intention-spec` skill), constraints, and what "good" looks like into one place the work can read.

2. **Design — write the spec.** Turn the vision into a one-paragraph build spec: what the first real artifact is, who it's for, what done looks like. Decide the approach once.

3. **Build — render a first version.** Use AI to make a real first version fast — a page, a draft, a track, a prototype. Set state first with the `vibe-align` skill. Rough and real beats polished and imagined.

4. **Automate (optional this cycle) — find the repeat.** If this goal involves a repeating job, identify the one loop worth wiring so it runs without the user. Not every first loop needs this; name it for next time if so.

5. **Compound — read the signal.** Ship the artifact to one real person or publish it small. Capture what came back and name the single lesson. Feed it into the next loop.

## Steps for the session

- Confirm the goal and run each move as a short back-and-forth, producing the artifact for that move before advancing.
- At Build, actually generate the first version with the user, don't just plan it.
- End by writing the next loop: what they keep, what changes, and the daily practice they're keeping.

## Output

For each move, the concrete artifact (context notes · spec · the built first version · optional automation note · the shipped result + lesson) and a one-line definition of the next loop.

## Boundaries

- The loop must end in a real action or artifact. If a session ends with only plans, it isn't a loop — say so.
- Keep scope to one cycle. Depth and breadth come from running more loops, not from over-building the first.
